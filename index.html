Introduction

Welcome to Game of Life - a JavaScript adaptation of Conway's Game of Life. Game of Life can be described as a mathematics-based game, where cells within the game will either live on or die in the next generation according to the number of neighbours that are alive.

Rules:
  • Any live cell with fewer than two live neighbours dies, as if caused by under-population
  • Any live cell with two or three live neighbours lives on to the next generation.
  • Any live cell with more than three live neighbours dies, as if by overcrowding.
  • Any dead cell with exactly three live neighbours becomes a live cell, as if by reproduction.

Following these rules, it is possible to create repeating patterns or constructors/factories. Try it yourself, or search online or through the help pages for defined examples.

Play/Edit Mode

Play Mode

Entering Play mode has several effects. It begins the execution of the game, killing and raising cells upon each new generation as according to the rules of Game of Life. An alive cell is indicated by a filled in square, whereas dead cells are indicated by squares which are not filled in. Be sure to keep an eye out for any changes between each generation. In addition, it disables the changing of cell state within the grid, as well as locking the settings. If you wish to makes changes to either the cells or the settings, you must exit Play mode by entering Edit mode.

The execution of the game can be quickened or slowed by changing the speed in the settings.

Edit Mode

While in Edit mode you can make changes to cells' state directly by clicking on the grid in the game view, or through the settings. An alive cell is indicated by a filled in square, whereas dead cells are indicated by squares which are not filled in. Click on a cell to toggle it's state from alive to dead, and vice versa. Once you are satisfied with the initial set up of your cells, enter Play mode to see how well your cells will fare throughout the generations.

Patterns

In Game of Life, there have been several defined patterns noted by the community as it exhibits a special behaviour. Groups of patterns covered in the following listing include still life, oscillators, spaceships, replicators, factories/guns and infinite life patterns.

  • Still Life
    A still life is called as such, due to their nature of remaining stable throughout generations. They do not spawn or kill off any cells or any surrounding cells in their vicinity.
  • Oscillators
    Oscillators will kill and spawn cells in a repetitive cycle which spans several generations. However, within this repeated cycle they can be considered to be stable, as they will reliably repeat this pattern.
  • Spaceships
    A stationary life isn't an end game for all cells. Spaceships will move across the grid, and will maintain some kind of form or shape which will repeat over few generations. They can continue to travel until they are interrupted by the presence of other cells.
  • Replicators
    Discovered relatively recently, a replicator will create an exact copy of itself while destroying the original in the process.
  • Factories/Gun
    These cells are positioned such that they will continually spawn a spaceship which will then glide across the grid.
  • Infinite Life
    Having an initial set up which uses an infinite life pattern will ensure that your cells will live on throughout the ages without fail, while also not being as stationary and unchanging as a still life.
Settings

Using the settings will let you alter some aspects of the game itself. Below is an explanation of the settings, how to change them and what values are allowed if applicable.

  • Columns
    This affects the grid. This specifies the number of columns to use in our grid. This can be changed through the plus or minus buttons, or by manual entering a number into the textbox. Only integers higher than 0 (non-inclusive) are accepted.

    WARNING: A higher value has an adverse affect on performance. Be sure to increment slowly to ensure smooth performance.
  • Rows
    This affects the grid. This specifies the number of rows to use in our grid. This can be changed through the plus or minus buttons, or by manual entering a number into the textbox. Only integers higher than 0 (non-inclusive) are accepted.

    WARNING: A higher value has an adverse affect on performance. Be sure to increment slowly to ensure smooth performance.
  • Speed
    This affects the execution of the game. This is used as a relative indicator to how quickly each generation should be processed, with a higher number indicating that it should take less time. This can be changed by using the slider control, or by manually entering into the textbox. Only integers between 0 (non-inclusive) and 100 (inclusive) are allowed.
  • Clear
    This affects the cells within the grid. This button sets all the cells within the grid to a dead state, allowing for a clean grid.
